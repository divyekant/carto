# Carto Glossary

Canonical definitions for terms used across the Carto codebase, documentation, and generated artifacts. Alphabetical order.

---

### .carto directory

**Definition:** A hidden directory at the project root where Carto stores local state, including the manifest file (SHA-256 hashes for incremental indexing) and any cached intermediate artifacts.

**Context:** Created automatically during `carto index`. Used by the manifest package for change detection.

**Not to be confused with:** The Carto binary itself, or the `carto` CLI command.

---

### .cursorrules

**Definition:** A skill file generated by the patterns package, targeting the Cursor IDE. Contains codebase architecture summaries, module descriptions, patterns, and index-maintenance instructions formatted for Cursor's context system.

**Context:** Written to the project root after the Skill Files phase of the pipeline. Content is bounded by `<!-- BEGIN CARTO INDEX -->` / `<!-- END CARTO INDEX -->` markers.

**Also known as:** Cursor rules file, skill file (when referring generically alongside CLAUDE.md).

---

### Atom

**Definition:** A concise, LLM-generated summary of a single code chunk. Atoms form layer 1a of the 7-layer context graph and are the fundamental unit of semantic indexing.

**Context:** Produced by the `atoms` package using the fast-tier LLM. Each chunk yields one atom. Atoms are stored in Memories and used as input for deep-tier analysis.

**Not to be confused with:** Chunks (the raw code segments that atoms summarize).

---

### Blueprint

**Definition:** A narrative description of the system's overall architecture, purpose, and design rationale. The blueprint is layer 4 of the 7-layer context graph, produced by the deep-tier LLM.

**Context:** Generated by the `analyzer` package during the Deep Analysis phase. Synthesizes information from atoms, wiring, and zones into a coherent architecture summary. Stored in Memories and surfaced in skill files.

---

### cliError

**Definition:** A structured error type defined in `errors.go` that carries a human-readable message (`msg`), a machine-readable error code (`code`), and a process exit code (`exit`). All CLI errors are either created as `cliError` instances or classified into one via `toCliError()`.

**Context:** Constructor functions create typed errors: `newConnectionError()`, `newAuthError()`, `newNotFoundError()`, `newConfigError()`. The `toCliError()` function wraps any untyped error as `GENERAL_ERROR` with exit code 1. Used by the envelope system and `runWithEnvelope()`.

**Not to be confused with:** Standard Go `error` interface. `cliError` is a Carto-specific wrapper that adds error codes and exit codes on top of the basic error string.

---

### Chunk

**Definition:** A discrete segment of source code extracted by the chunker, typically corresponding to a function, method, class, or logical block as identified by Tree-sitter AST parsing.

**Context:** Produced by the `chunker` package during the Chunk+Atoms phase. Each chunk is sent to the fast-tier LLM for atom extraction. Chunk boundaries are language-aware when Tree-sitter support is available; otherwise, line-based fallback is used.

**Not to be confused with:** Atoms (the LLM-generated summaries of chunks).

---

### CLAUDE.md

**Definition:** A skill file generated by the patterns package, targeting Claude-based AI assistants (Claude Code, Claude in IDE, etc.). Contains codebase architecture, modules, patterns, and active index-maintenance workflow instructions.

**Context:** Written to the project root after the Skill Files phase. Content between `<!-- BEGIN CARTO INDEX -->` / `<!-- END CARTO INDEX -->` markers is managed by Carto; content outside is preserved.

**Also known as:** Skill file (when referring generically alongside .cursorrules).

---

### Error Code

**Definition:** A machine-readable string that classifies the type of CLI error. Appears in the `code` field of JSON error envelopes. The five error codes are: `GENERAL_ERROR` (unhandled), `NOT_FOUND` (resource missing), `CONFIG_ERROR` (invalid/missing config), `CONNECTION_ERROR` (unreachable service), and `AUTH_FAILURE` (bad/missing API key).

**Context:** Defined as constants in `errors.go`. Each error code maps to a corresponding exit code (1-5). Used by agents and scripts for programmatic error classification -- more specific than exit codes alone.

**Not to be confused with:** Exit codes (numeric process return values). Error codes are string identifiers in the JSON envelope; exit codes are integers returned to the shell.

---

### Exit Code

**Definition:** A numeric value (0-5) returned by the Carto CLI process to indicate the outcome of a command. Exit code 0 means success. Non-zero codes indicate specific failure categories.

**Context:** Defined in `helpers.go` and `errors.go`. The mapping is: 0 = success, 1 = general error, 2 = not found, 3 = config error, 4 = connection error, 5 = auth failure. Exit codes are set by `runWithEnvelope()` or by direct `os.Exit()` calls. Shell scripts can branch on `$?` to handle different failure categories.

**Not to be confused with:** Error codes (string identifiers in JSON envelopes). Exit codes are for the shell; error codes are for JSON consumers.

---

### Deep Tier

**Definition:** The expensive LLM tier used for cross-component analysis. Runs fewer calls with a more capable model (default: `claude-opus-4-6`) to produce wiring maps, zone groupings, and the blueprint.

**Context:** Configured via `CARTO_DEEP_MODEL`. Used by the `analyzer` package during the Deep Analysis phase (phase 4). Produces layers 2-4 of the context graph.

**Also known as:** Deep model, deep analysis tier.

**Not to be confused with:** Fast tier (high-volume, per-chunk processing).

---

### Fast Tier

**Definition:** The high-throughput LLM tier used for per-chunk atom extraction. Runs many concurrent calls with a smaller, faster model (default: `claude-haiku-4-5-20251001`).

**Context:** Configured via `CARTO_FAST_MODEL`. Concurrency controlled by `CARTO_MAX_CONCURRENT` (default: 10). Used by the `atoms` package during the Chunk+Atoms phase (phase 2). Produces layer 1a of the context graph.

**Also known as:** Fast model, atom extraction tier.

**Not to be confused with:** Deep tier (fewer calls, more capable model).

---

### Gold Brand Palette

**Definition:** The Carto CLI's visual identity color scheme, consisting of five named colors used across all terminal output: Brand Gold (#d4af37) for primary accents, headers, and active states; Stone (#78716c) for de-emphasis and neutral text; Amber (#F59E0B) for warnings; Rose (#F43F5E) for errors and destructive actions; and Emerald (#10B981) for success indicators.

**Context:** Defined in `branding.go` (brand constants) and `helpers.go` (ANSI escape codes). The CLI maps these hex colors to the closest ANSI equivalents: `\033[33m]` for gold, `\033[38;5;249m]` for stone, `\033[38;5;214m]` for amber, `\033[31m]` for red, `\033[32m]` for green. The palette replaced the earlier indigo/cyan color scheme in v1.2.0 and applies to all human-mode output.

**Not to be confused with:** The Web UI's Tailwind CSS color palette, which can render the exact hex values. Terminal ANSI codes are approximations.

---

### History

**Definition:** Git history data extracted for each file, including commit frequency, recent changes, and author information. History forms layer 1b of the 7-layer context graph.

**Context:** Produced by the `history` package during the History+Signals phase (phase 3). Provides temporal context -- which files change often, who works on what, and how the codebase evolves.

---

### JSON Envelope

**Definition:** The standard output format for all Carto CLI commands in JSON mode. A success envelope is `{"ok": true, "data": <payload>}` written to stdout. An error envelope is `{"ok": false, "error": "<message>", "code": "<ERROR_CODE>"}` written to stderr. This contract matches the Memories CLI convention.

**Context:** Implemented in `output.go` via `writeEnvelope()` and `writeEnvelopeHuman()`. JSON mode is triggered by the `--json` flag, by TTY auto-detection (non-terminal stdout), or by the `runWithEnvelope()` command runner. The envelope provides a consistent parsing contract for CI pipelines, AI agents, and scripts -- consumers check the `ok` field rather than parsing command-specific output formats.

**Not to be confused with:** Raw NDJSON output from `carto export` (which streams individual records, not envelopes). The `--json` flag on `export` switches from raw NDJSON to envelope mode with a summary.

---

### Layer

**Definition:** One of the 7 levels in the Carto context graph. Each layer captures a different dimension of codebase understanding, from individual code summaries to system-wide architecture.

**Context:** The 7 layers are: (0) Map, (1a) Atoms, (1b) History, (1c) Signals, (2) Wiring, (3) Zones, (4) Blueprint. Stored in Memories with layer-specific source tags. Retrieval tiers (mini/standard/full) control how many layers are included in query results.

---

### Manifest

**Definition:** A file containing SHA-256 hashes for every indexed source file. Used by the manifest package to detect changes between indexing runs and enable incremental indexing.

**Context:** Stored in the `.carto` directory. On each indexing run, current file hashes are compared against the manifest. Only files with changed or missing hashes are reprocessed. Deleting the manifest forces a full re-index.

**Not to be confused with:** Module manifests like `go.mod` or `package.json` (which define project boundaries for module detection).

---

### Memories

**Definition:** An external vector storage service that Carto uses to persist and retrieve all indexed data. Memories provides semantic search over stored layers.

**Context:** Carto communicates with Memories via a REST client (the `storage` package). Configured via `MEMORIES_URL` (default: `http://localhost:8900`) and `MEMORIES_API_KEY`. Required for all indexing and querying operations. The docker-compose setup includes a Memories instance.

**Not to be confused with:** In-memory caching or local storage.

---

### NDJSON

**Definition:** Newline-Delimited JSON. A text format where each line is a complete, self-contained JSON object. Used by Carto for streaming data in the `export` command, the audit log file, and the `import` command's input format.

**Context:** The `export` command streams index data as NDJSON (one memory record per line) for piping to files or other tools. The `import` command reads NDJSON from stdin. The audit log (`CARTO_AUDIT_LOG`) stores one JSON event per line. Each NDJSON line is independently parseable, making the format suitable for streaming, appending, and line-by-line processing with tools like `jq`.

**Not to be confused with:** JSON arrays (`[{...}, {...}]`) or the JSON envelope format (`{"ok": true, "data": ...}`). NDJSON has no wrapping structure -- each line stands alone.

---

### Module

**Definition:** A project boundary detected by the scanner based on the presence of manifest files (e.g., `go.mod`, `package.json`, `Cargo.toml`). Each module represents a self-contained unit within a larger codebase.

**Context:** The `scanner` package identifies modules during the Scan phase (phase 1). Module information is used to scope indexing and organize results. Viewable via `carto modules`.

---

### Pattern

**Definition:** A recurring design approach, architectural convention, or coding practice identified across the codebase during deep analysis. Patterns are included in generated skill files.

**Context:** Identified by the `analyzer` package and surfaced via the `patterns` package. Examples: "dependency injection via interfaces," "event-driven architecture," "repository pattern." Viewable via `carto patterns`.

**Not to be confused with:** The `patterns` package (which generates skill files, not just pattern lists).

---

### runWithEnvelope

**Definition:** A centralized command runner function defined in `errors.go` that executes a command function, writes the result via the JSON envelope system, logs an audit event, and exits with the appropriate exit code on error. It is the standard "run a command" wrapper that new commands should adopt.

**Context:** Signature: `runWithEnvelope(cmd *cobra.Command, humanFn func(data any), fn func() (any, error))`. On success, it calls `writeEnvelopeHuman()` with the data and the human renderer, then logs an "ok" audit event. On error, it classifies the error via `toCliError()`, prints a colored error in human mode, writes the error envelope in JSON mode, logs an "error" audit event, and calls `os.Exit()` with the exit code from the `cliError`. This eliminates duplicated error handling across commands.

**Not to be confused with:** `writeEnvelopeHuman()` (which only handles output formatting, not error classification or exit codes). `runWithEnvelope` is the full lifecycle wrapper; `writeEnvelopeHuman` is the lower-level output function.

---

### Signal

**Definition:** A unit of external context collected from a source plugin during the History+Signals phase. Signals enrich the index with information beyond the code itself -- issues, documentation, design decisions, chat discussions.

**Context:** Produced by the `signals` package using the unified Source interface. Signals form layer 1c of the context graph. Each signal has a source tag identifying its origin (e.g., `github`, `jira`, `notion`).

**Not to be confused with:** Unix signals (process signals). In Carto, "signal" always refers to external context data.

---

### Skill File

**Definition:** A generated file (CLAUDE.md or .cursorrules) that provides AI assistants with structured codebase context and active index-maintenance instructions. The primary output artifact of a Carto indexing run.

**Context:** Generated by the `patterns` package during the Skill Files phase (phase 6). Contains architecture summaries, module lists, patterns, and curl-based instructions for writing back to the index. See also: CLAUDE.md, .cursorrules.

---

### Source

**Definition:** A plugin that fetches external data to enrich the Carto index. Sources implement a unified interface and are configured per-project. Available sources: Git, GitHub, Jira, Linear, Notion, Slack, PDF, Web.

**Context:** Managed by the `signals` package. Configured via environment variables (e.g., `GITHUB_TOKEN`, `JIRA_URL`) and per-project settings. Sources run concurrently during the History+Signals phase. A source failure is non-fatal -- the pipeline continues with remaining sources.

**Also known as:** Source plugin, signal source.

**Not to be confused with:** "Source code" (the code being indexed). In Carto, "source" refers to external data providers unless explicitly qualified.

---

### Source Tag

**Definition:** A string label attached to stored data in Memories that identifies its origin layer and source. Used for scoped retrieval and updates.

**Context:** Format: `carto/<project>/<module>/layer:<layer-name>`. Examples: `carto/myapp/api/layer:atoms`, `carto/myapp/core/layer:blueprint`. Source tags enable the storage package to update specific layers without affecting others.

---

### SSE (Server-Sent Events)

**Definition:** A server-push protocol used by the Carto REST API to stream real-time pipeline progress to clients during indexing operations.

**Context:** Implemented in the `server` package. The Web UI consumes SSE events to display live progress (phase transitions, file counts, errors, completion). v1.0.0 fixed SSE event naming for consistency across the API.

**Not to be confused with:** WebSockets (bidirectional). SSE is server-to-client only.

---

### Tier (Retrieval)

**Definition:** A retrieval granularity level that controls how much context is returned from a query. Carto supports three tiers: mini (~5KB), standard (~50KB), and full (~500KB).

**Context:** Implemented in the `storage` package. The tier determines which layers of the context graph are included in query results. Mini returns high-level summaries; full returns the complete context graph. Selected via CLI flags or API parameters.

**Not to be confused with:** LLM tiers (fast/deep), which refer to the models used during indexing, not retrieval.

---

### TTY Auto-Detection

**Definition:** A mechanism in the Carto CLI that automatically determines whether to emit JSON envelope output or human-readable colored output based on whether stdout is connected to a terminal. Implemented via `isJSONMode()` in `output.go` using `golang.org/x/term.IsTerminal()`.

**Context:** The detection priority is: (1) `--json` flag explicitly set -- JSON mode. (2) `--pretty` flag explicitly set -- human mode (overrides `--json`). (3) Neither flag -- fall back to TTY check: non-terminal stdout triggers JSON; terminal stdout triggers human. This means piping output to another program, redirecting to a file, or capturing from an AI agent subprocess automatically triggers JSON without any flags. The design eliminates the need for CI pipelines and agents to remember `--json`.

**Not to be confused with:** The `--json` flag itself (which is an explicit override). TTY auto-detection is the fallback behavior when no explicit flag is set.

---

### Tree-sitter

**Definition:** An incremental parsing library used by Carto for language-aware AST-based code chunking. Tree-sitter produces concrete syntax trees that the chunker uses to split code at semantically meaningful boundaries (functions, classes, methods).

**Context:** Used by the `chunker` package. Requires CGO for compilation (the build needs a C compiler). If Tree-sitter does not support a file's language, the chunker falls back to line-based splitting.

**Not to be confused with:** The Go AST parser (which is Go-specific). Tree-sitter is a multi-language parser.

---

### Wiring

**Definition:** A map of cross-component dependencies and interaction patterns across the codebase. Wiring forms layer 2 of the 7-layer context graph.

**Context:** Produced by the `analyzer` package using the deep-tier LLM during the Deep Analysis phase. Identifies which modules call each other, shared interfaces, data flow paths, and integration points. Used as input for zone detection and blueprint generation.

---

### Zone

**Definition:** A business domain grouping that clusters related modules and components by functional area. Zones form layer 3 of the 7-layer context graph.

**Context:** Produced by the `analyzer` package using the deep-tier LLM. Zones group modules into higher-level business domains (e.g., "authentication," "billing," "data pipeline"). Used as input for blueprint generation and surfaced in skill files.

**Not to be confused with:** Modules (which are project boundaries based on manifest files). Zones are semantic groupings that may span multiple modules.
